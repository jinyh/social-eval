# SocialEval 开发路线图（重排版）

**创建日期**：2026-04-15  
**约束来源**：`docs/requirements/SocialEval-requirements-v0.4.md`、`CLAUDE.md`、`docs/architecture/20260414_ADR-001_evaluation-framework-v2.md`

---

## 一、重排原则

本路线图不再直接沿用“先 API、后补冲突”的方式，而是采用：

1. **先修契约，再做功能**
2. **先打通闭环，再补外围**
3. **代码 / schema / 文档 / 测试必须同版**

---

## 二、阶段划分

### 阶段 0：契约对齐

目标：统一当前活动框架 law v2.0 的代码契约。

重点：
- 对齐 `schema_v2.json` 与 `law-v2.0-20260413.yaml`
- 让 loader 支持 v2，并保留 v1 兼容
- 统一 prompt / precheck prompt 渲染
- 补齐 v2 测试基线

交付标准：
- v2 配置可加载
- v2 prompt 可渲染
- v2 测试通过

### 阶段 1：平台骨架与认证

目标：建立可用的 FastAPI 平台骨架与统一鉴权机制。

重点：
- `src/api/main.py`
- Session 登录态
- API Key 认证
- 角色依赖
- 用户 / 邀请 / 健康检查路由

### 阶段 2：评审闭环 MVP

目标：打通上传到 AI 评审入库的最小闭环。

流程：
上传论文 → 预处理 → precheck → 多模型评分 → reliability → 状态回写 → 最小内部结果输出

### 阶段 3：报告与专家复核

目标：补全业务闭环。

重点：
- 内部 / 公开报告
- JSON / PDF 导出
- 低置信度入复核队列
- 专家指派、通知、提交
- 报告版本化重生成

### 阶段 4：状态机、审计与恢复

目标：补足生产级运行能力。

重点：
- 状态机
- 审计日志
- 管理员恢复 / 关闭
- 批量任务统计

### 阶段 5：前端交付

目标：按优先级逐步交付前端。

顺序：
1. 编辑端
2. 投稿人端
3. 专家端
4. 管理员端


---

## 四、关键实施口径

### 1. 当前活动框架

- 当前活动法学框架：`configs/frameworks/law-v2.0-20260413.yaml`
- `law_v1.yaml` 仅作为兼容 / 回归测试样本保留

### 2. 认证口径

- Web：Session / Cookie
- 第三方系统：API Key
- 不以 JWT 作为主认证方案

### 3. 报告建设口径

- 先做“可用导出”
- 后做“视觉美化”

---

## 五、阶段 0 之后再处理的内容

以下内容明确不放入阶段 0：
- FastAPI 主入口与路由
- Celery 编排
- 报告生成
- 专家复核
- 前端实现

这样可以确保第一阶段只解决“当前系统定义本身不成立”的问题。
