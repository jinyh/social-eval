## 1. 当前总体结论

目前系统已经从“仅本地开发骨架”推进到了“可试点演示、可登录、可上传、可评测、可生成报告、可分配专家、可做专家复核、可做基础管理、可进行试点部署访问”的阶段。

已经打通的核心闭环是：

`登录 -> 上传稿件 -> 预检 -> 多模型评测 -> 可靠性判断 -> 生成公开/内部报告 -> 专家分配 -> 专家复核 -> 导出报告`

但当前仍然属于“试点可用版”，不是“完整正式生产版”。  
尤其在以下方面还没有完全收口：

- 投稿人视图和内部评审视图之间的边界还需要进一步严格拆分
- 投稿人下载 PDF 与内部报告导出的内容层级还需要继续区分
- 完整运营后台、正式监控告警、正式备份恢复体系还没有完全落地

## 7. 关于你刚刚提的事情，我的理解与后续处理方向

你刚刚提出的要求，本质上是要把“投稿人可见内容”和“内部评审可见内容”彻底拆开。

我当前的理解是：

### 7.1 投稿者 submitter 应该看到什么

投稿者应主要看到：

- 稿件基础信息
- 当前评测状态
- 综合结果或最终结论
- 必要的简要摘要
- 必要的修改建议或结论性反馈
- 简洁版 PDF 下载

投稿者不应直接看到：

- 过细的维度级 AI 内部分析
- 后台模型比较细节
- 证据摘录原文堆叠
- 专家复核内部过程信息
- 原始报告 JSON

### 7.2 审稿/复核侧应看到什么

编辑和专家才更适合看到：

- 逐维度评分
- AI 分析细节
- 证据摘录
- 置信度信息
- 多轮复核信息
- 专家复核记录
- 必要时的内部报告导出

### 7.3 这件事之后我打算怎么做

我准备按下面顺序推进：

#### 第一步：重划角色可见边界

- 把 submitter 页面从“详细展示型公开报告”改成“简洁结果页”
- 去掉 submitter 页面上的原始 JSON 暴露
- 去掉 submitter 页面上不该出现的后台 AI 评审细节

#### 第二步：拆分导出策略

- submitter 下载：输出“简洁版公开 PDF”
- editor / expert：保留“内部详细报告”
- 让不同角色拿到的报告内容和详略层级明确分离

#### 第三步：前端界面分层

- submitter：结果卡片 + 简洁摘要 + 结论建议
- editor：任务全貌 + 队列 + 分配 + 内部分析
- expert：逐维度细节 + 复核录入

#### 第四步：补测试

- 为 submitter 视图新增“不暴露内部字段”的测试
- 为 PDF 导出新增“角色分层内容正确”的测试
- 为 API/前端分别补可见性边界测试
