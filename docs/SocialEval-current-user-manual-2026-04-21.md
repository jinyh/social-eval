## 1. 先看结论

当前这个仓库已经具备四类用户角色：

- 投稿者 `submitter`
- 编辑 `editor`
- 专家 `expert`
- 管理员 `admin`

系统已经打通的主流程是：

`登录 -> 上传论文 -> 自动发起评测 -> 生成公开/内部报告 -> 低置信度任务进入复核队列 -> 编辑分配专家 -> 专家提交复核 -> 报告更新`

但需要特别说明：

- 当前主仓库里的前端是“可用版/演示版”，不是完整运营版
- 投稿者前端相对完整
- 编辑、专家、管理员前端目前偏简化
- 有些后端能力已经存在，但前端还没有做成完整操作界面

因此，这份手册会把内容分成两类：

- `前端可直接使用`
- `后端已支持，但当前前端未完全提供`

## 2. 当前系统能力总览

### 2.1 认证方式

当前系统支持两种认证方式：

- Web 登录：Session / Cookie
- API 调用：`X-API-Key`

当前前端已提供的是：

- 邮箱 + 密码登录
- 退出登录

当前后端已提供但前端未做完整入口的是：

- 邀请码激活账号
- API Key 签发

### 2.2 支持的论文文件格式

当前允许上传的格式只有：

- `pdf`
- `docx`
- `txt`

不支持的格式会直接被拒绝，例如：

- `doc`
- 其他未知扩展名

### 2.3 评测触发方式

在当前主仓库里：

- `上传文件后会立即创建任务并自动触发评测`
- `前端没有“开始评测”按钮`

这意味着当前系统不是“先上传、后手动开始”的模式，而是“上传即入评测流程”。

### 2.4 报告分层

当前系统有两类报告：

- `public`：公开报告
- `internal`：内部报告

区别如下：

- 公开报告给投稿者等非内部评审角色查看
- 内部报告给编辑、管理员，以及被分配任务的专家查看
- 内部报告比公开报告包含更多 AI 评分细节

### 2.5 当前真实前端状态

基于当前 `/home/xiwen/social-eval/src/web/src/App.tsx`，各角色界面状态如下：

| 角色 | 当前前端状态 | 说明 |
|---|---|---|
| 投稿者 | 相对完整 | 可上传、看论文列表、看结果、下载简洁版 PDF |
| 编辑 | 简化版 | 可看复核队列、选专家、查看内部报告 JSON |
| 专家 | 很简化 | 可看自己的 review 列表，并一键提交标准复核 |
| 管理员 | 简化版 | 可建邀请、看用户列表 |

## 3. 通用使用说明

### 3.1 登录

当前 Web 端登录流程：

1. 打开前端页面
2. 输入邮箱和密码
3. 点击登录
4. 登录成功后，系统根据角色进入不同工作台

当前登录页对应代码：

- `src/web/src/App.tsx`

后端登录接口：

- `POST /api/auth/login`

### 3.2 退出

前端右上角有退出按钮，对应：

- `POST /api/auth/logout`

### 3.3 健康检查

系统提供无需登录的健康检查接口：

- `GET /api/health`

返回示例：

```json
{
  "status": "ok",
  "service": "socialeval"
}
```

### 3.4 当前常见状态说明

后端任务状态目前主要包括：

- `pending`：任务已创建，尚未开始
- `processing`：系统正在处理
- `reviewing`：已进入专家复核阶段
- `completed`：任务完成
- `recovering`：任务失败，等待管理员处理
- `closed`：任务已被管理员关闭

补充说明：

- 论文还会带 `precheck_status`
- 如果 `precheck_status = reject`，表示论文在预检阶段未通过，不会进入正式维度评分

注意：

当前投稿者前端的状态文字映射并没有完整覆盖所有后端状态，因此在某些情况下，界面展示可能和后端原始状态不完全一致。

## 4. 投稿者使用手册

### 4.1 投稿者当前能做什么

当前主仓库中，投稿者前端可直接完成：

- 登录系统
- 上传论文
- 查看自己上传的论文列表
- 选择某篇论文查看结果
- 查看公开报告中的总分、综合结论、各维度一句话总结
- 查看专家最终结论（如果有）
- 下载简洁版 PDF 报告

相关前端代码：

- `src/web/src/App.tsx` 中 `SubmitterDashboard`

相关后端接口：

- `POST /api/papers`
- `GET /api/papers`
- `GET /api/papers/{paper_id}/status`
- `GET /api/papers/{paper_id}/report`
- `GET /api/papers/{paper_id}/export/simple`

### 4.2 投稿者标准操作流程

#### 步骤 1：登录

使用投稿者账号登录系统。

#### 步骤 2：上传论文

在“投稿者工作台”中：

1. 点击文件选择框
2. 选择 `pdf/docx/txt` 文件
3. 点击“上传论文”

上传成功后：

- 系统会返回一个 `paper_id`
- 同时自动创建并触发评测任务

#### 步骤 3：查看论文列表

上传后，论文会出现在“我的论文”列表中。

当前列表可见信息包括：

- 论文标题
- 上传文件名
- 当前处理状态

#### 步骤 4：查看结果

点击某一篇论文后，右侧“评价结果”区域会显示：

- 论文标题
- 总分
- 综合结论
- 各维度分数
- 各维度一句话总结
- 专家评语（如果复核后有最终专家结论）

#### 步骤 5：下载简洁版报告

点击“下载报告”按钮后，系统会调用：

- `GET /api/papers/{paper_id}/export/simple`

导出内容是简洁版 PDF，主要包括：

- 论文标题
- 综合结论
- 总分
- 各维度分数
- 一句话总结
- 专家最终结论

简洁版报告默认不包含：

- AI 详细分析
- 证据引用
- 置信度指标
- 专家姓名

### 4.3 投稿者当前看不到什么

基于当前设计，投稿者不直接查看：

- 内部报告
- 模型级别打分
- 专家逐维度修改理由
- 复核任务队列
- 用户管理能力

### 4.4 投稿者当前限制

当前主仓库里的投稿者前端还没有：

- 拖拽上传
- 多文件批量上传界面
- 模型选择界面
- 评测轮数设置
- 历史版本报告切换
- 删除稿件功能

补充说明：

- 后端其实已经支持批量上传接口 `POST /api/papers/batch`
- 但当前主仓库前端没有提供对应入口

## 5. 编辑使用手册

### 5.1 编辑当前能做什么

当前主仓库中，编辑前端可直接完成：

- 查看复核队列
- 获取专家列表
- 为队列中的任务分配专家
- 打开某一任务的内部报告

相关前端代码：

- `src/web/src/App.tsx` 中 `EditorDashboard`

相关后端接口：

- `GET /api/reviews/queue`
- `GET /api/users/experts`
- `POST /api/reviews/{task_id}/assign`
- `GET /api/papers/{paper_id}/internal-report`

### 5.2 编辑标准操作流程

#### 步骤 1：登录编辑账号

登录后进入 `Editor Dashboard`。

#### 步骤 2：查看待复核任务

系统会从：

- `GET /api/reviews/queue`

加载低置信度任务或已请求人工复核的任务。

当前队列项主要包含：

- `task_id`
- `paper_id`
- 论文标题
- 低置信度维度列表

#### 步骤 3：选择专家

系统会从：

- `GET /api/users/experts`

读取专家列表。当前前端是单选下拉框。

#### 步骤 4：分配专家

点击按钮后，当前前端会对“队列中的第一条任务”调用：

- `POST /api/reviews/{task_id}/assign`

注意：

- 当前前端不是完整的任务选择式分配面板
- 更准确地说，它是“最小可用演示版”
- 它默认分配当前队列第一项，而不是成熟后台里常见的可视化任务卡片操作

#### 步骤 5：查看内部报告

点击某条任务旁边的 `Open internal report` 后，前端会调用：

- `GET /api/papers/{paper_id}/internal-report`

然后把结果直接以 JSON 文本形式显示在页面右侧。

内部报告中当前可看到：

- 维度级 AI 均分
- 标准差
- 是否高置信度
- `model_scores`
- `evidence_quotes`
- `analysis`
- 专家复核记录

### 5.3 编辑当前的实际限制

当前主仓库前端还没有为编辑提供：

- 中文化完整后台
- 多任务切换式管理面板
- 报告可视化页面
- 内部报告 PDF/JSON 导出按钮
- 多专家批量勾选分配界面
- 任务筛选、搜索、分页

但后端已经有一部分能力：

- 队列查询
- 专家分配
- 内部报告读取

因此当前编辑角色属于：

- `后端能力可用`
- `前端后台仍是简化版`

## 6. 专家使用手册

### 6.1 专家当前能做什么

当前主仓库前端里，专家界面只提供最小功能：

- 查看自己被分配到的 review 列表
- 提交一份标准复核

相关前端代码：

- `src/web/src/App.tsx` 中 `ExpertDashboard`

相关后端接口：

- `GET /api/reviews/mine`
- `POST /api/reviews/{review_id}/submit`

### 6.2 专家标准操作流程

#### 步骤 1：登录专家账号

登录后进入 `Expert Dashboard`。

#### 步骤 2：查看待处理 review

前端会调用：

- `GET /api/reviews/mine`

当前列表展示的信息比较少，仅包含：

- `review_id`
- `task_id`
- `status`

#### 步骤 3：提交复核

点击 `Submit standard review` 后，当前前端会直接提交一组固定结构的评论。

当前前端提交内容的特点是：

- 默认覆盖六个维度
- `ai_score` 固定为 `70`
- `expert_score` 固定为 `72`
- `reason` 固定为 `Frontend submission`

这意味着：

- 当前专家前端并不是完整的人工逐项评审表单
- 它更像是一个“演示式标准提交按钮”

### 6.3 专家当前真正具备的后端能力

后端实际上支持：

- 逐维度提交复核意见
- 每条意见包含：
  - `dimension_key`
  - `ai_score`
  - `expert_score`
  - `reason`

也就是说：

- 后端支持“认真做专家复核”
- 但当前主仓库前端还没有把这个能力完整做成可编辑表单

### 6.4 专家当前限制

当前主仓库前端还没有：

- 逐维度查看详细 AI 依据的专家工作台
- 自定义评分输入框
- 自定义复核理由输入框
- 查看自己已提交意见的完整回显界面

所以专家端当前属于：

- `接口闭环已存在`
- `前端工作台还是极简版`

## 7. 管理员使用手册

### 7.1 管理员当前能做什么

当前主仓库前端里，管理员界面可以：

- 创建邀请
- 查看用户列表

相关前端代码：

- `src/web/src/App.tsx` 中 `AdminDashboard`

相关后端接口：

- `GET /api/users`
- `POST /api/users/invitations`

后端另外还支持：

- 重试失败任务
- 关闭任务

对应接口：

- `POST /api/admin/tasks/{task_id}/retry`
- `POST /api/admin/tasks/{task_id}/close`

### 7.2 管理员标准操作流程

#### 步骤 1：登录管理员账号

登录后进入 `Admin Dashboard`。

#### 步骤 2：创建邀请

页面中可填写：

- 邀请邮箱
- 角色

可邀请角色包括：

- `submitter`
- `editor`
- `expert`
- `admin`

提交后，前端会调用：

- `POST /api/users/invitations`

#### 步骤 3：查看用户目录

当前页面会显示已有用户的：

- 显示名或邮箱
- 角色

### 7.3 邀请后的激活方式

这里有一个很重要的当前限制：

- 后端已经支持邀请激活
- 但当前主仓库前端没有专门的“接受邀请/激活账号”页面

后端激活接口是：

- `POST /api/auth/invitations/accept`

需要提交：

- `token`
- `display_name`
- `password`

所以当前真实情况是：

- 管理员可以创建邀请
- 受邀用户可以通过后端接口激活
- 但当前主仓库 Web 前端没有完整激活页面

### 7.4 管理员后端已支持但前端未提供的能力

后端已经有，但当前前端没有按钮或面板的能力包括：

- 失败任务重试
- 任务关闭
- 审计相关操作链路

因此管理员端当前也属于：

- `接口能力部分完整`
- `前端后台仍偏基础`

## 9. 当前主要局限与使用建议

### 9.1 当前最成熟的是投稿者主流程

如果你要对外演示，当前最稳的是：

- 投稿者登录
- 上传论文
- 自动评测
- 查看简洁结果
- 下载简洁版 PDF

### 9.2 编辑、专家、管理员更适合内部试用，不适合直接当成完整产品后台

原因是：

- 编辑前端还是简化面板
- 专家前端还是演示式提交
- 管理员前端只有基础邀请和用户查看

### 9.3 如果你要给真实用户培训，建议按“当前可用”和“暂未前端化”分开讲

推荐培训口径：

- 投稿者：可直接按现有界面使用
- 编辑：能完成最基本分配与查看
- 专家：当前前端只适合演示，不适合作为正式评审工作台
- 管理员：当前可做账户管理，运维控制更多依赖接口和后端操作
