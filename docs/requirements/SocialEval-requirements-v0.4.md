# SocialEval 系统需求文档

## 背景与目标

当前传统学术评价面临"不可能三角"困境：标准不一（依赖主观经验）、效率低（评审周期漫长）、成本高（专家调度成本巨大）。本项目旨在构建一套以"自主知识体系"为评价底层逻辑的 AI 辅助学术评价系统，以法学论文评审为切入点，最终拓展到人文社科各学科。

**项目名称**：SocialEval — 面向自主知识体系的人工智能辅助学术评价系统

**主要目标**：
- 用 AI 辅助实现高质量、标准统一的学术论文初审
- 以"六维评价框架"量化评分，并提供可解释的评价依据
- 通过多模型并发验证保证评分可靠性
- 提供 Web 界面和 API 接口，支持人工和系统集成两种使用方式

---

## 用户角色与核心诉求

| 角色 | 核心诉求 |
|------|----------|
| **投稿人/研究者** | 接收邀请后上传论文、查看评审进度、获取最终公开报告 |
| **期刊编辑委员** | 批量初审、查看完整 AI 评分报告、指定专家、查看部分已提交复核结果、决定是否送外审 |
| **领域专家（评审者）** | 查看被分配论文的完整 AI 评分报告、进行人工复核、对各维度填写修改意见 |
| **系统管理员** | 管理邀请、配置评价维度/知识体系、查看统计数据、处理失败任务恢复 |

### 用户开通方式

- **U1.1** v1 采用**全角色邀请制**；投稿人、编辑、专家、管理员均不支持开放自注册
- **U1.2** 受邀用户通过邀请链接完成账号激活与密码设置
- **U1.3** Web 端用户认证采用 **Session**
- **U1.4** 第三方系统调用开放 API 时采用 **API Key**

## 功能需求

### F1. 文档摄取与预处理

- **F1.1** 支持上传 PDF、Word（.docx）、TXT 格式的学术论文
- **F1.2** 预处理分两步：
  - **噪声去除**：自动去除页眉、页脚、致谢等与评价无关内容
  - **参考文献独立提取**：参考文献区块不作为噪声丢弃，而是单独提取存储，供「现状洞察度」评价维度在评分时调用
- **F1.3** 提取论文正文结构（摘要、引言、主体、结论）；
  对于**非标准排版**论文（无明确章节标签），系统自动通过字体规律、编号规律等特征检测章节边界；
  无法可靠识别时降级为"全文分段"模式，并在评审报告中附加结构识别警告标记
- **F1.4** 输出标准化预处理结果，包含：正文核心、参考文献列表（独立字段）、结构识别状态
- **F1.5** v1 **默认不支持扫描版 PDF OCR**；对疑似扫描版 PDF，在上传或预处理阶段明确提示当前不支持，并引导重新上传可解析文本版文件
- **F1.6** `MinerU` 作为后续 OCR/复杂文档解析扩展方向记录，不纳入 v1 交付范围

### F2. 知识体系配置

- **F2.1** 评价框架通过外部 YAML/JSON 配置文件定义，系统动态加载
- **F2.2** 配置内容包括：
  - 学科名称
  - 评价维度列表，每个维度包含：名称、描述、**权重**（0–1 浮点数，所有维度之和须为 1.0）、提示词模板
  - 一致性阈值（标准差上限，可覆盖全局默认值）
- **F2.3** 内置法学评价框架（六维，默认等权），作为默认配置：
  1. **问题创新性**（Problem Originality）
  2. **现状洞察度**（Literature Insight）
  3. **理论建构力**（Theoretical Construction）
  4. **逻辑严密性**（Logical Coherence）
  5. **学术共识度**（Scholarly Consensus）
  6. **前瞻延展性**（Forward Extension）
- **F2.4** 管理员可通过后台界面上传/切换不同学科的知识体系配置文件
- **F2.5** 配置文件支持版本管理（保存历史版本，支持回滚）

### F3. AI 评价引擎

- **F3.1** 支持调用多家商业 AI 模型（OpenAI GPT-4o、Anthropic Claude、DeepSeek 等），通过统一接口抽象切换
- **F3.2** 针对每个评价维度独立构造提示词，AI 逐维度输出**结构化 JSON**，字段包含：
  ```json
  {
    "dimension": "问题创新性",
    "score": 82,
    "evidence_quotes": ["论文原文引用片段1", "..."],
    "analysis": "AI 分析说明文本"
  }
  ```
- **F3.3** 支持"多模型并发评价"模式：同一论文同一维度同时提交给 N 个模型，收集各模型评分
- **F3.4** v1 多模型并发数配置规则：
  - 默认值为 **3**
  - 上限为 **5**
- **F3.5** 支持"智能体研讨会"模式（高级，可配置选项）：
  - 角色设置：主持人、批判者、总结者等多角色
  - **终止条件**（按优先级依次判断）：
    1. 连续两轮评分变化幅度 < 配置阈值（提前收敛，可配置，默认 3 分）
    2. 达到最大轮数上限（可配置，默认 5 轮）后强制结束
  - 强制结束时未收敛的维度标记为「研讨未收敛」，自动进入人工介入队列
- **F3.6** 所有 AI 调用记录（输入/输出/时间戳/模型名称/耗时）须持久化存储，供审计追溯
- **F3.7** AI 调用的完整原始输入与完整原始输出作为长期保留的审计对象，不仅保留结构化评分结果

### F4. 可靠性验证层

- **F4.1** 多模型评分收集完成后，自动计算各维度的均值、标准差
- **F4.2** 一致性阈值判断：
  - 全局默认值：标准差 ≤ 5 分（管理员可在系统设置中修改全局默认值）
  - 各学科可在知识体系 YAML 中单独配置阈值，覆盖全局默认值
  - 达标则标记"高置信度"
- **F4.3** 未达标（分歧较大）的维度/论文自动标记"低置信度"，进入专家复核队列
- **F4.4** 报告中须展示：各模型原始评分、均值 ± 标准差、置信度标签
- **F4.5** 专家复核后，专家评分与 AI 评分**并列存储**，均纳入最终报告展示，不互相覆盖
- **F4.6** v1 不生成融合后的单一"最终裁决分"；AI 评分与专家评分作为两套并列结果保留

### F5. 专家复核工作流

- **F5.1** 低置信度论文自动进入复核队列；编辑也可对高置信度论文**手动追加专家复核**
- **F5.2** 专家通过系统账号登录后，可查看分配给自己的待复核论文列表
- **F5.3** 专家可查看论文全文（或预处理后的净化版本）及 AI 评分报告（含各模型原始评分）
- **F5.4** 专家可对每个维度修改评分，并填写修改理由（文本，必填）
- **F5.5** **专家分配机制**：编辑在论文详情页手动指定一名或多名平台专家账号；凡进入复核队列且已完成专家指定，系统自动向被指定专家发送邮件通知（含论文摘要和 AI 评分摘要）
- **F5.6** 专家可通过邮件中的链接直接跳转到复核页面（需已登录或跳转后提示登录）
- **F5.7** 复核结果需记录：专家身份、操作时间、各维度修改前/后评分、修改理由
- **F5.8** 多专家复核采用**并行模式**
- **F5.9** 当一篇论文分配给多名专家时，需在**所有已分配专家均提交**后，才视为该论文的专家复核完成
- **F5.10** 编辑可在专家**未提交前**撤回或重分配专家；专家一旦提交，其复核记录不撤回
- **F5.11** 专家提交后不可自行直接修改；如需修改，须由编辑或管理员退回后重提
- **F5.12** 仅编辑和管理员可触发"退回重提"
- **F5.13** 编辑和管理员可在多人复核过程中查看部分已提交的专家结果，但这些内容仅内部可见，并须标记为"复核未完成"
- **F5.14** 专家退回重提时，系统生成新的复核结果版本与新的报告版本，历史版本全部保留，不覆盖旧版本

### F6. 评审报告生成

- **F6.1** 系统区分两类报告：
  - **公开结果报告**：供投稿人查看
  - **内部完整报告**：供编辑、被分配专家、管理员查看
- **F6.2** 公开结果报告包含以下字段：

  | 字段 | 说明 |
  |------|------|
  | 综合加权总分 | 各维度均值按 YAML 配置权重加权求和（0-100 分） |
  | 六维评分 | 各维度分值 |
  | 六维雷达图 | 以各维度分值绘制（可视化） |
  | 每维简评 | 每个维度对应的简短文字评语 |
  | 总评 | 面向投稿人的整体文字评价 |
  | 专家意见摘要 | 如有专家复核，仅展示摘要，不展示专家身份 |

- **F6.3** 内部完整报告包含以下字段：

  | 字段 | 说明 |
  |------|------|
  | 六维 AI 评分 | 各维度：均值 ± 标准差，置信度标签 |
  | 综合加权总分 | 各维度均值按 YAML 配置权重加权求和（0-100 分） |
  | AI 评价依据 | 每维度展示少量代表性论文原文引用片段 + AI 分析说明 |
  | 多模型一致性数据 | 各模型原始评分列表、均值、标准差 |
  | 六维雷达图 | 以各维度均值绘制（可视化） |
  | 专家复核意见 | 如有：专家评分（并列展示，不覆盖 AI 评分）、修改理由、复核时间 |

- **F6.4** 多专家场景下，报告中并列展示各专家意见摘要，不做统一裁决
- **F6.5** 公开结果仅在**任务完全结束后**对投稿人可见；不提供 AI 完成后的临时公开初稿
- **F6.6** 每次生成报告都创建一个**报告快照版本**；当前有效版本用于对外展示，历史版本保留用于追溯
- **F6.7** 支持导出 PDF 格式报告（供人工阅读）
- **F6.8** 支持导出 JSON 格式原始数据（供第三方系统集成）

### F7. Web 用户界面

- **F7.1** **投稿人界面**：论文上传、进度查看、最终公开报告下载
  - 投稿人仅可查看公开结果报告，不展示 AI 各模型原始评分及多模型一致性数据
  - 投稿人状态统一映射为：待处理、评审中、需重新上传、已完成
- **F7.2** **编辑界面**：论文列表管理、批量初审视图、单篇内部完整报告详情、专家指定操作、失败任务处理跟踪
- **F7.3** **专家界面**：复核任务列表、含 AI 完整评分的评分修改页面
- **F7.4** **管理员界面**：邀请管理、用户管理、知识体系配置上传、一致性阈值全局设置、系统统计看板、失败任务恢复操作
- **F7.5** 所有界面需要身份认证（登录/权限控制）

### F8. API 接口

- **F8.1** 提供 RESTful API，支持第三方系统集成；v1 提供标准 API 能力，但**不做现成期刊系统的预置对接**
- **F8.2** 核心 API 端点：
  - `POST /api/papers` — 上传论文，返回任务 ID
  - `GET /api/papers/{id}/status` — 查询评审进度
  - `GET /api/papers/{id}/report` — 获取公开结果报告（JSON）
  - `GET /api/papers/{id}/internal-report` — 获取内部完整报告（JSON）
  - `POST /api/papers/batch` — 批量提交论文任务
- **F8.3** API 需要 Token 认证（API Key 机制）
- **F8.4** `GET /api/papers/{id}/internal-report` 仅编辑、被分配专家、管理员可访问
- **F8.5** 批量任务完成通知在 v1 **仅支持轮询**；不要求回调 URL 机制

### F9. 任务状态与异常恢复

- **F9.1** 单篇论文任务采用"**业务主状态 + 失败原因/子状态**"模型
- **F9.2** 内部主状态至少包括：
  - `pending`
  - `processing`
  - `reviewing`
  - `recovering`
  - `completed`
  - `closed`
- **F9.3** `reviewing` 用于表示专家复核阶段，不与 `processing` 混用
- **F9.4** `recovering` 用于表示管理员正在对技术性失败任务进行手动恢复
- **F9.5** `closed` 为终态；被管理员标记关闭后，不再重新打开。如需重做，应创建新任务或重新上传文件
- **F9.6** 失败应按阶段区分，至少包括：
  - 预处理失败
  - AI 评审失败
  - 报告生成失败
  - 待人工重试
- **F9.7** AI 调用失败时自动重试（最多 3 次）；仍失败则进入"待人工重试"
- **F9.8** 仅**技术性失败**可由管理员手动重试，例如模型调用失败、队列异常、报告生成失败
- **F9.9** 文件损坏、格式非法、疑似扫描版 PDF 等输入类问题，直接标记为输入失败，并提示用户重新上传或更换文件，不进入管理员重试队列
- **F9.10** 管理员最小恢复动作包括：
  - 查看失败原因
  - 手动重试当前阶段
  - 标记关闭
- **F9.11** 批量任务作为汇总容器，状态由子任务聚合生成，并至少提供总数、完成数、失败数等统计字段

### F10. 数据留存与审计

- **F10.1** 系统对以下对象保留完整原始记录：
  - AI 调用输入与输出
  - 预处理结果
  - 专家复核明细
  - 报告快照版本
- **F10.2** 数据库保存元数据与索引；完整大文本原始内容保存于对象存储或文件型存储
- **F10.3** 审计日志最小字段集包括：
  - 操作者
  - 时间
  - 对象
  - 动作
  - 结果
- **F10.4** 内部完整报告访问属于必须审计的事件；至少记录查看人、查看时间、对应论文或任务对象、报告类型、访问结果
- **F10.5** 投稿人查看公开结果报告不纳入内部访问审计范围
- **F10.6** 每次报告生成、重生成、专家重提后再生成，均创建新的报告版本；历史版本长期保留
- **F10.7** `v1` 默认长期保留原始大文本记录，不提供手动清理或保留期管理界面

## 非功能需求

### 性能
- 单篇论文（2万字以内）AI 评审完成时间：≤ 10 分钟
- API 批量任务支持并发处理，单次批量上限 ≥ 100 篇
- Web 界面核心页面加载时间：≤ 3 秒

### 可靠性
- 系统可用性：≥ 99%（工作时间）
- AI 调用失败时自动重试（最多 3 次），失败后进入可恢复的异常处理流程并通知管理员
- 报告、复核、审计日志均需支持历史版本追溯，不因后续操作覆盖旧记录

### 安全性
- 所有接口 HTTPS 加密传输
- Web 用户身份认证采用 Session
- API Key 权限隔离
- 论文数据允许云端存储及调用外部 AI API（数据不做本地隔离要求）
- 投稿人不可访问 AI 原始调用数据和多模型分歧数据
- 内部完整报告仅编辑、被分配专家、管理员可访问
- 内部完整报告访问需纳入审计日志

### 可扩展性
- 知识体系通过外部配置文件管理，新增学科无需改代码
- AI 模型接口通过统一抽象层管理，新增模型只需扩展配置
- 未来可扩展扫描版 PDF OCR / MinerU 文档解析通道
- 未来可扩展对象存储生命周期管理与审计归档策略
- 后期可将各模块拆分为独立微服务
