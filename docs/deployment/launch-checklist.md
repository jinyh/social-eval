# 文科论文评价系统上线检查清单与切换计划

**版本**：2026-04-16  
**目标**：将当前 `production-launch-readiness` 交付项落到可执行的生产上线流程  
**适用范围**：单机 MVP 首次上线、版本切换、紧急回滚

---

## 1. 上线目标与退出标准

本次上线的目标不是“服务启动成功”，而是以下链路全部可用：

1. 用户可通过正式域名访问前端
2. Session 登录与权限控制正常
3. 投稿上传、异步评测、报告读取链路正常
4. 数据库存储、Redis 队列、对象存储、SMTP 均可工作
5. `/api/health` 返回 `200`
6. 冒烟链路全部通过且无阻塞告警

若以上任一项不满足，则上线不算完成。

---

## 2. 角色分工建议

- 发布负责人：负责时间窗口、变更确认、是否回滚的最终决策
- 应用负责人：负责镜像、迁移、应用日志、业务冒烟
- 平台负责人：负责主机、DNS、TLS、反向代理、监控、备份
- 业务验收人：负责以真实账号验证关键角色流程

首次上线不建议单人同时承担全部角色，至少需要一名旁路验证人。

---

## 3. 切换时间线

### T-7 天到 T-3 天：资源与依赖准备

- 确认云主机、系统盘、数据盘、对象存储 bucket 已创建
- 确认 PostgreSQL、Redis、SMTP、对象存储白名单和网络策略已放通
- 确认正式域名与 TLS 证书已准备
- 确认 `.env` 生产值已收集完毕并放入密钥管理系统
- 确认 build 机可访问 Debian 软件源或内网镜像

### T-3 天到 T-1 天：预发布演练

- 用同一份部署脚本在预发布环境走完整流程
- 执行数据库迁移并记录耗时
- 执行一次完整冒烟
- 验证对象存储上传与导出
- 验证 SMTP 发信
- 整理上线窗口中的实际命令清单

### T-1 天：冻结与备份

- 代码冻结，除 blocker 外不再接收变更
- 导出 PostgreSQL 备份或创建磁盘快照
- 记录当前稳定版本 commit / 镜像 tag
- 复核回滚包、回滚命令和负责人联系方式

### T 日：正式切换

- 发布负责人宣布开始上线窗口
- 停止非必要后台操作，避免上线期间写入冲突
- 按第 4 节执行部署步骤
- 按第 5 节执行冒烟
- 观察至少 30 分钟后再宣布上线完成

### T+1 小时到 T+24 小时：加强观察

- 关注 API 5xx、Celery 队列堆积、SMTP 失败率、数据库磁盘空间
- 核对新上传论文是否持续成功
- 对外通知仅在观察期内无异常后发出

---

## 4. 正式上线执行清单

### 4.1 变更前检查

- `.env` 已确认 `APP_ENV=production`
- `SECRET_KEY` 已替换默认值
- `ALLOWED_ORIGINS` 已配置正式前端来源
- `SESSION_COOKIE_SECURE=true`
- `SESSION_COOKIE_DOMAIN` 与正式域名一致
- AI Provider、SMTP、对象存储密钥已填入
- Nginx、DNS、TLS 已就绪

### 4.2 部署执行

1. 拉取目标版本代码
2. 渲染并验证编排配置

```bash
docker compose -f docker-compose.prod.yml config
```

3. 构建镜像

```bash
docker compose -f docker-compose.prod.yml build api worker frontend
```

4. 启动依赖

```bash
docker compose -f docker-compose.prod.yml up -d postgres redis
```

5. 执行数据库迁移

```bash
docker compose -f docker-compose.prod.yml run --rm api uv run alembic upgrade head
```

6. 启动应用

```bash
docker compose -f docker-compose.prod.yml up -d api worker frontend nginx
```

7. 验证健康检查

```bash
curl -fsS https://app.socialeval.example/api/health
```

### 4.3 首个管理员初始化

若生产库中还没有管理员：

- 按部署说明中的一次性脚本创建首个 `admin`
- 立即完成首次登录
- 用该管理员邀请 `submitter`、`editor`、`expert`

---

## 5. 冒烟测试清单

至少验证以下路径：

1. 正式域名可访问首页
2. `/api/health` 返回 `200`
3. 管理员可以登录
4. 管理员可以创建邀请
5. 受邀用户可以完成激活
6. 投稿人可以上传样例论文
7. 任务可以被 Worker 正常消费并完成
8. 投稿人可以读取公开报告
9. 编辑可以读取内部报告与复核队列
10. 对象存储中出现 `uploads/` 与 `exports/` 对象
11. 若发生专家分配，SMTP 邮件能送达测试邮箱

建议记录：

- 每一步开始与结束时间
- 负责执行的人
- 实际请求 ID、论文 ID、任务 ID
- 失败时对应日志位置

---

## 6. 回滚触发条件

满足任一条件应立即评估回滚，不要拖到业务全面受影响：

- `/api/health` 持续返回 `503` 超过 5 分钟
- 登录链路异常，管理员或投稿人无法建立 Session
- 新上传论文无法进入或完成异步任务
- 报告接口持续报错
- 数据库迁移失败或造成关键表读写异常
- Worker 堆积明显上升且无法恢复
- 对象存储或 SMTP 关键依赖配置错误，导致主流程不可用

---

## 7. 回滚步骤

### 7.1 应用层回滚

1. 暂停新一轮变更操作
2. 记录故障时间点、报错请求、任务 ID
3. 切回上一个稳定 commit 或镜像 tag
4. 重新启动 `api`、`worker`、`frontend`、`nginx`
5. 再次执行健康检查与最小冒烟

### 7.2 数据层回滚

若数据库迁移已经执行且与旧版本不兼容：

1. 停止应用写入
2. 使用上线前快照或逻辑备份恢复 PostgreSQL
3. 校验关键表结构与数据量
4. 恢复旧版本应用
5. 重新执行最小冒烟

### 7.3 回滚后动作

- 保留故障日志、数据库错误、Celery 错误堆栈
- 明确根因、修复方案、再次上线窗口
- 未完成根因分析前，不要重复上线

---

## 8. 上线后 24 小时观察项

- API 5xx 比例是否稳定
- 登录失败率是否异常
- Celery 队列深度是否持续增长
- PostgreSQL CPU、连接数、磁盘空间是否异常
- Redis 内存与持久化状态是否正常
- 对象存储写入是否持续成功
- SMTP 是否存在退信或认证失败

---

## 9. 上线结论记录模板

建议在每次上线后补一份简短记录，至少包含：

- 上线时间窗口
- 目标版本 commit / tag
- 执行人 / 验收人
- 数据库迁移版本
- 冒烟结果
- 是否发生回滚
- 已知遗留问题

这份记录建议保存在团队运维文档或变更系统中，避免只留在聊天消息里。
